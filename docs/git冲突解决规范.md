# 冲突解决指南

> 在 PR 工作流中，冲突通常发生在：**你的分支和 master 都改了同一个文件的同一个地方**。  
> 不要慌，按下面的步骤来，5 分钟搞定。


## 一、冲突长什么样？

当你的分支和 master 都改了同一个文件时，GitHub 会显示类似这样的内容：

```
<<<<<<< feat/add-login (你的改动)
登录按钮
=======
注册按钮
>>>>>>> master
```

### 含义说明

| 标记 | 含义 |
|------|------|
| `<<<<<<<` | 你的改动开始 |
| `=======` | 分隔线，上面是你的，下面是 master 的 |
| `>>>>>>> master` | master 的改动结束 |

### 怎么解决

```
你需要：删除标记 → 只保留最终想要的内容
```

**示例：**

```diff
- <<<<<<< feat/add-login
  登录按钮
- =======
- 注册按钮
- >>>>>>> master
```

**最终变成：**

```
登录按钮
```

> 如果两个都要，就合并成 `登录 / 注册按钮`


## 二、各角色推荐方式

| 角色 | 推荐方式 | 理由 |
|------|----------|------|
| **翻译** | 方式一：GitHub 网页端解决 | 改的是文案，网页直接改最快 |
| **美术** | 方式一：GitHub 网页端解决 | 不涉及代码逻辑，网页操作最安全 |
| **开发** | 方式二：命令行解决 | 可能需要本地测试验证代码能否运行 |


## 三、方式一：GitHub 网页端解决（推荐翻译/美术）

### 适用场景
- 冲突内容不多（几行文案/代码）
- 不涉及复杂逻辑

### 操作步骤

**第 1 步：在 PR 页面点击 Resolve conflicts**

```
打开 PR 页面
→ 看到红色提示 "This branch has conflicts that must be resolved"
→ 点击 Resolve conflicts 按钮
```

**第 2 步：手动解决冲突**

```
页面会列出所有冲突文件
→ 找到冲突区域（有 <<<<<<< 标记的地方）
→ 删除标记，保留最终想要的内容
→ 删除多余内容
```

**第 3 步：标记已解决并提交**

```
解决完所有冲突文件后
→ 点击 Mark as resolved（标记为已解决）
→ 点击 Commit merge（提交合并）
→ PR 自动更新，冲突消失 ✅
```


## 四、方式二：命令行解决（推荐开发）

### 适用场景
- 冲突文件较多
- 需要本地测试解决后代码是否正常运行

### 操作步骤

**第 1 步：拉取最新的 master 到你的分支**

```bash
# 切换到你的功能分支
git checkout feat/xxx

# 拉取最新的 master 代码
git fetch origin master

# 把 master 合并到你的分支（会触发冲突）
git merge origin/master
```

**第 2 步：查看哪些文件冲突了**

```bash
git status

# 输出示例：
# both modified:   src/App.js
# both modified:   src/locales/zh-CN.json
```

**第 3 步：手动解决冲突**

用代码编辑器打开冲突文件：

```
搜索 <<<<<<< HEAD
找到冲突区域
删除标记，保留最终想要的内容
保存文件
```

**第 4 步：标记已解决并提交**

```bash
# 添加所有已解决的文件
git add .

# 提交合并
git commit -m "merge: 解决与 master 的冲突"

# 推送到远程（PR 会自动更新）
git push origin feat/xxx
```

**第 5 步：回到 PR 页面确认**

```
PR 页面会显示 "All checks have passed"
→ 冲突已解决 ✅
→ 继续等待审核或合并
```


## 五、方式三：VS Code 可视化解决（推荐不熟悉命令的人）

### 操作步骤

**第 1 步：在 VS Code 中打开冲突文件**

```
文件内容会显示冲突区域，并有可视化按钮
```

**第 2 步：选择保留哪个版本**

```
┌─────────────────────────────────────────────────────────────┐
│  <<<<<<< HEAD (当前分支 - 你的改动)                        │
│  登录按钮                                                  │
│  =======                                                  │
│  注册按钮                                                  │
│  >>>>>>> master (master 分支的内容)                        │
│                                                             │
│  ┌──────────┐ ┌────────────┐ ┌───────────┐               │
│  │ Accept    │ │ Accept     │ │ Accept    │               │
│  │ Current   │ │ Incoming   │ │ Both      │               │
│  │ (保留你的) │ │ (保留master)│ │ (两个都要)│               │
│  └──────────┘ └────────────┘ └───────────┘               │
└─────────────────────────────────────────────────────────────┘
```

| 按钮 | 效果 |
|------|------|
| **Accept Current** | 保留你的改动（删除 master 的） |
| **Accept Incoming** | 保留 master 的改动（删除你的） |
| **Accept Both** | 两个都保留 |

**第 3 步：保存文件 → 提交 → 推送**

```bash
git add .
git commit -m "merge: 解决与 master 的冲突"
git push origin feat/xxx
```


## 六、如何提前避免冲突？

| 习惯 | 做法 | 效果 |
|------|------|------|
| **每天拉取 master** | 开始工作前先 `git pull origin master` | 保持本地 master 最新 |
| **及时合并 master** | 开发过程中定期把 master 合并进分支 | 小步快跑，冲突小 |
| **沟通修改范围** | 团队内提前说好要改哪些文件 | 从源头避免 |
| **PR 不要拖太久** | 分支生命周期不超过 2 天 | 降低冲突概率 |
| **不要改别人的文件** | 翻译只改文案文件，美术只改资源文件 | 职责分明 |


## 七、各角色处理建议速查

| 遇到冲突 | 怎么办 |
|----------|--------|
| **翻译** | 用 GitHub 网页点击 `Resolve conflicts` 直接改文案 |
| **美术** | 同上，网页解决 |
| **开发** | 命令行或 VS Code 解决，解决完本地测试一下 |
| **不确定怎么改** | 在群里 @ 相关人确认，不要凭感觉改 |
| **冲突太多** | 放弃当前分支，重新从 master 建分支，手动复制改动 |


## 八、极端情况：放弃分支重新来

如果冲突太多、太乱，可以直接放弃当前分支，重新建一个：

```bash
# 1. 保存当前改动到临时文件（备份）
git stash

# 2. 切回 master，拉取最新
git checkout master
git pull origin master

# 3. 删除旧分支（本地 + 远程）
git branch -D feat/xxx
git push origin --delete feat/xxx

# 4. 新建分支，重新开始
git checkout -b feat/xxx

# 5. 把备份的改动应用回来
git stash pop
```

> 💡 这种方式适合 **开发还没正式开始** 或 **改动内容很少** 的情况。


## 九、一句话总结

| 情况 | 一句话建议 |
|------|-----------|
| 你是翻译/美术 | 网页点 `Resolve conflicts`，删标记保留想要的文案 |
| 你是开发 | 命令行 `git merge origin/master` 解决后本地测试再 push |
| 完全不知道怎么改 | 在 PR 评论区 @ 相关人，说“这里冲突了，谁来帮我看看” |
